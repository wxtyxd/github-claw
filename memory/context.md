# memory/context.md - 项目背景与核心决策

> 本文档记录 github-claw 项目的核心决策、架构理念、设计选择和长期愿景。
> 这是理解"为什么这样设计"的关键文档。

---

## 🎯 项目使命

**目标**：让 GitHub Copilot 从一个"对话机器"升级为"长期协作者"

**核心问题**：
- 为什么 AI 助手每次对话都像"失忆"一样从零开始？
- 如何让 AI 记住我的工作方式、决策逻辑和已有经验？
- 如何将 AI 的输出从一次性的回答升级为持久的知识资产？

**解决方案**：
通过 GitHub 仓库作为"持久化大脑"，让 Copilot 能够：
- 跨对话继承角色和工作方式
- 积累工作模式和最佳实践
- 系统化地追踪任务进度
- 持续改进工作流程

---

## 🏗️ 架构设计

### 核心原则

#### 1. 文件即记忆
- 不依赖外部数据库或复杂系统
- 使用 Markdown 作为通用格式（易于阅读、易于版本控制）
- 每个文件都有明确的职责和生命周期

#### 2. 人类优先
- 所有文档都应能被人类直接阅读和编辑
- 避免过度设计，优先实用性
- 决策过程应该透明，而不是隐藏在代码或配置中

#### 3. 渐进式演化
- 系统从简单开始，根据实际使用逐步完善
- 每个新模式都通过实战验证后才被固化
- 定期回顾和调整，不怕改变

### 文件层级

```
github-claw/
├── 核心规范层（长期保留 1+ 年）
│   ├── AGENTS.md           # Copilot 工作手册
│   ├── README.md           # 项目入口
│   └── memory/
│       ├── context.md      # 本文件 - 决策和背景
│       └── patterns.md     # 已验证的工作模式
│
├── 进度追踪层（保留 1-2 月）
│   ├── PROGRESS.md         # 当前任务状态
│   └── memory/daily/       # 日常日志
│
└── 待建设
    └── memory/snippets/    # 代码和文档片段库
```

## 📋 核心决策

### 决策 #1: 为什么选择 GitHub 仓库而不是其他存储方案？

**时间**：2026-06-06  
**背景**：
- 需要一个持久化的存储方案来保存 Copilot 的"记忆"
- 已有多个候选方案：本地文件、云笔记、数据库、文本编辑器

**选项**：
1. **GitHub 仓库** ✓ 选中
   - 优点：版本控制、历史追溯、易于共享、Copilot 天然支持、完全免费
   - 缺点：需要网络、不适合实时协作

2. 本地文件系统
   - 优点：离线工作、速度快
   - 缺点：易丢失、无版本控制、难以跨设备同步

3. 云笔记（Notion/语雀）
   - 优点：易于使用、支持富文本
   - 缺点：API 支持有限��Copilot 集成度差

4. 数据库（SQLite/PostgreSQL）
   - 优点：结构化、查询灵活
   - 缺点：过度工程化、违反"人类优先"原则

**选择**：GitHub 仓库  
**理由**：
- 最符合"文件即记忆"和"人类优先"的设计理念
- Copilot 天然集成，可直接读写文件
- 版本控制对于理解工作演化至关重要
- 公开仓库便于分享工作方式

**结果**：
- ✅ 所有文档都通过 Git 管理，有完整的修改历史
- ✅ 可以在任何 Copilot 对话中无缝访问
- ✅ 决策和工作流程一目了然

---

### 决策 #2: 为什么分离"规范层"和"追踪层"？

**时间**：2026-06-06  
**背景**：
- AGENTS.md、patterns.md 是长期稳定的规范
- PROGRESS.md、daily logs 是不断变化的进度信息
- 需要平衡文档的稳定性和灵活性

**选项**：
1. **分离存储** ✓ 选中
   - 长期记忆（1+ 年保留）：AGENTS.md, context.md, patterns.md
   - 进度追踪（1-2 月保留）：PROGRESS.md, daily/

2. 全部混在一起
   - 优点：简单统一
   - 缺点：日志会淹没规范，查阅困难

3. 使用多个仓库
   - 优点：完全隔离
   - 缺点：过度复杂，维护成本高

**选择**：分离存储  
**理由**：
- AGENTS.md 是规范，应该稳定可靠，很少修改
- daily logs 每天都在变化，不应该"污染"核心规范
- 新对话启动时，应该快速加载规范，而不是被日志淹没

**结果**：
- ✅ Copilot 启动快速：先读 AGENTS.md（稳定）→ 再查 PROGRESS.md（最新）
- ✅ 日志可以大胆记录和更新，不怕破坏规范
- ✅ 1-2 月后可以归档 daily logs，不影响仓库大小

---

### 决策 #3: 为什么必须有"收尾流程"？

**时间**：2026-06-06  
**背景**：
- 如果没有系统化的收尾，知识会散落，进度会丢失
- 每个对话都是独立的，需要主动连接它们

**选项**：
1. **强制收尾清单** ✓ 选中
   - 优点：确保知识不丢失、进度持续更新、下次对话无缝接棒
   - 缺点：增加对话时间

2. 自愿记录
   - 优点：灵活、不强制
   - 缺点：容易遗忘、知识散落

3. 完全自动化
   - 优点：无需手动
   - 缺点：难以实现、容易出错

**选择**：强制收尾清单  
**理由**：
- 收尾是"系统化工作流程"的关键环节
- 规范化的流程确保一致性
- 短期（5-10 分钟）的收尾投入，换来长期的知识积累

**结果**：
- ✅ 每次对话结束都会自动更新进度和日志
- ✅ 下次对话启动时，可以立即看到上次的工作成果
- ✅ 逐步建立完整的工作历史档案

---

### 决策 #4: 为什么使用 Markdown 而不是其他格式？

**时间**：2026-06-06  
**背景**：
- 需要一个易于阅读、易于版本控制的文档格式

**选项**：
1. **Markdown** ✓ 选中
   - 优点：轻量、易读、易写、GitHub 原生支持、可转换为 HTML
   - 缺点：不支持复杂排版

2. Org-mode
   - 优点：功能强大、支持时间管理
   - 缺点：生态小、学习曲线陡

3. HTML
   - 优点：功能完整
   - 缺点：难以编辑、版本控制混乱

**选择**：Markdown  
**理由**：
- Copilot 对 Markdown 天然理解，无需解析
- GitHub 完美渲染，阅读体验好
- 足以承载 80% 的文档需求

**结果**：
- ✅ 所有文档都是 .md 格式，易于阅读和编辑
- ✅ GitHub 上的渲染效果美观
- ✅ 易于在 Copilot 中处理

## 🎓 从失败中学习

### 已识别的风险

1. **日志堆积风险**
   - 风险：日志越来越多，查阅困难
   - 缓解：每月做一次总结，归档旧日志

2. **规范过时风险**
   - 风险：AGENTS.md 跟不上实际工作变化
   - 缓解：每月回顾一次，根据反馈及时更新

3. **孤岛化风险**
   - 风险：知识只在 Copilot 脑子里，人类无法接手
   - 缓解：所有关键决策都记录在 context.md，用人类语言

4. **过度工程化风险**
   - 风险：规范变得太复杂，难以遵循
   - 缓解：KISS 原则，优先实用性

## 🤝 协作模式

### 人 - Copilot 协作框架

```
用户（人）
  │
  ├─ 提出需求/指令
  │   ↓
  └─ 验证工作质量
      ↑
  Copilot（AI）
  ├─ 理解背景（阅读 AGENTS.md）
  ├─ 查看进度（阅读 PROGRESS.md）
  ├─ 执行工作
  ├─ 更新文档（memory/）
  └─ 提交代码（git commit）
```

**信任建立**：
- Copilot 严格遵守 AGENTS.md 的规范
- 每个决策都有记录和理由
- 工作进度透明且易于审查
- 定期寻求用户反馈并改进
